# Section 01: Interaction State System

**Status:** Not Started
**Goal:** Every widget can query `ctx.is_hot()`, `ctx.is_active()`, `ctx.is_focused()` without
tracking any mouse state itself. The framework computes Hot automatically from layout geometry,
manages Active via explicit set/clear, and routes Focus exclusively. No widget ever stores
`is_hovered: bool` again.

**Context:** Currently every interactive widget (Button, Toggle, Dropdown, Checkbox, Slider,
TextInput) independently tracks its own `hovered: bool` state via synthetic
`HoverEvent::Enter/Leave` events generated by `InputState` (in
`oriterm_ui/src/input/routing.rs`). Each container also has a `ContainerInputState`
(`oriterm_ui/src/widgets/container/mod.rs`) that tracks hover and capture among its children.
This leads to duplicated logic across 6 leaf widgets + `ContainerInputState`, inconsistent
enter/leave behavior, and missed leave events on fast mouse movement. Druid solved this by
making hover state framework-managed and automatic.

**Reference implementations:**
- **Druid** `druid/src/contexts.rs`: `is_hot()`, `is_active()`, `set_active(true)`, `is_focused()`
  — three orthogonal booleans that compose cleanly
- **GTK4** `GtkEventControllerMotion`: `contains-pointer` (subtree) vs `is-pointer` (direct)
  — correct nested hover model

**Depends on:** Nothing (foundation section).

**WARNING — Co-implementation with Section 03**: Hot state computation (`update_hot_path`)
requires the event propagation pipeline (Section 03) to know which widgets the pointer
is over and to deliver `HotChanged` before `MouseMove`. If Section 01 is implemented alone
without Section 03, hot tracking will work in isolation tests but cannot be wired into the
real event flow. Plan accordingly: either implement Sections 01+02+03 as a single batch, or
implement Section 01 with a clear integration seam that Section 03 hooks into.

**Module declaration sync point**: `pub mod interaction;` must be added to
`oriterm_ui/src/lib.rs` as part of this section. This is not automatic — the module will
be invisible to the crate until declared.

## 01.1 InteractionState Type

**File(s):** `oriterm_ui/src/interaction/mod.rs` (new module)

Define the per-widget interaction state owned by the framework.

- [x] Add `pub mod interaction;` to `oriterm_ui/src/lib.rs` (without this declaration the
  module is invisible to the rest of the crate)
- [x] Create `oriterm_ui/src/interaction/` module directory
- [x] Create `oriterm_ui/src/interaction/mod.rs` re-exporting `InteractionState`
  and `InteractionManager`. Declare sub-modules: `mod manager;`, `pub mod lifecycle;`,
  `mod parent_map;`. Re-export `build_parent_map` from `parent_map` at the module root.
  The interaction directory will contain: `mod.rs`, `manager.rs`, `lifecycle.rs`,
  `parent_map.rs`, `tests.rs`. (`parent_map.rs` is a separate file rather than inlined
  in `manager.rs` to keep `manager.rs` under the 500-line limit.)
- [x] Define `InteractionState` struct:
  ```rust
  /// Per-widget interaction state, managed by the framework.
  /// Widgets query this via context methods — they never set it directly.
  pub struct InteractionState {
      /// True when the pointer is over this widget or any descendant.
      /// Equivalent to GTK4's `contains-pointer`.
      hot: bool,
      /// True when the pointer is directly over this widget, not a descendant.
      /// Equivalent to GTK4's `is-pointer`.
      hot_direct: bool,
      /// True when the widget has captured mouse events (mouse-down without
      /// mouse-up). While active, the widget receives all mouse events
      /// regardless of pointer position.
      active: bool,
      /// True when this widget has keyboard focus.
      focused: bool,
      /// True when this widget or any descendant has keyboard focus.
      focus_within: bool,
      /// True when the widget is disabled (events are not routed).
      disabled: bool,
  }
  ```
- [x] Implement accessors: `is_hot()`, `is_hot_direct()`, `is_active()`, `is_focused()`,
  `has_focus_within()`, `is_disabled()`
- [x] Implement `InteractionState::new()` (all false) and `InteractionState::disabled()`
- [x] **Disabled flag population**: The `disabled` field in `InteractionState` is set by
  `InteractionManager::set_disabled(widget_id, disabled)`. This must be called whenever a
  widget's disabled state changes. Currently each widget owns its own `disabled: bool` (Button,
  Toggle, Checkbox, Slider, TextInput, Dropdown — all 6 interactive leaf widgets carry it).
  After Section 08 migration, disabled state is reported via `Widget::is_disabled() -> bool`
  called during layout/registration, and `InteractionManager::set_disabled()` is called from
  the framework when building the widget tree. For Section 01 purposes, expose
  `set_disabled(&mut self, id: WidgetId, disabled: bool)` on `InteractionManager` and leave
  the call-site wiring to Section 08. Calling `set_disabled` also queues a
  `LifecycleEvent::WidgetDisabled` event (see 01.3) so widgets can perform cleanup
  (e.g., clearing pressed or drag state).

## 01.2 InteractionManager

**File(s):** `oriterm_ui/src/interaction/manager.rs`

Centralized manager that maintains `InteractionState` for every widget in the tree.

- [x] Define `InteractionManager`:
  ```rust
  pub struct InteractionManager {
      /// Per-widget interaction state, keyed by WidgetId.
      states: HashMap<WidgetId, InteractionState>,
      /// Currently hot widget stack (from root to deepest).
      hot_path: Vec<WidgetId>,
      /// Currently active widget (only one at a time).
      active_widget: Option<WidgetId>,
      /// Currently focused widget (only one at a time).
      focused_widget: Option<WidgetId>,
      /// Pending lifecycle events to deliver on next frame.
      pending_events: Vec<LifecycleEvent>,
      /// child → parent map built from the last layout pass.
      /// Used to walk ancestors for focus_within updates.
      parent_map: HashMap<WidgetId, WidgetId>,
  }
  ```
- [x] `register_widget(&mut self, widget_id: WidgetId)`:
  Insert a new `InteractionState::new()` entry. Called on `WidgetAdded` lifecycle.
  Idempotent — if already registered, no-op.
- [x] `deregister_widget(&mut self, widget_id: WidgetId)`:
  Remove the widget from `states`, `hot_path`, and clear it from `active_widget`/
  `focused_widget` if it was active/focused. Generates `ActiveChanged(false)` or
  `FocusChanged(false)` lifecycle events if clearing active/focused state.
- [x] `update_hot_path(&mut self, new_path: &[WidgetId])`:
  Accept a pre-computed widget ancestor path (ordered root-to-leaf) produced by the
  Section 02 hit tester. Compare with previous `hot_path`. Generate `HotChanged` events for
  widgets that entered or left the hot path.
  - Last element in path: `hot = true, hot_direct = true`
  - All preceding elements: `hot = true, hot_direct = false`
  - Widgets in old `hot_path` but not in new path: `hot = false`
  - **Do not** accept a `LayoutNode` + `Point` pair and walk the tree internally — that
    responsibility belongs to the Section 02 widget hit tester which also applies Sense filtering.
  - Widgets with `Sense::none()` must never appear in the path passed to this method;
    the Section 02 hit tester filters them before calling `update_hot_path`.
- [x] **Widget-path function**: The `update_hot_path` API implies a new function (distinct from
  the existing `layout_hit_test` in `oriterm_ui/src/input/hit_test.rs`) that returns the full
  ancestor chain, not just the deepest hit. Name it `layout_hit_test_path` and define it in
  `oriterm_ui/src/input/hit_test.rs` alongside `layout_hit_test`. Return type:
  `Vec<WidgetId>` ordered root-to-leaf (or empty if no hit). Section 02 may refine this
  to filter by Sense; for now define the signature here so Section 01 tests can drive it.

  **WARNING — Implementation complexity**: `layout_hit_test_path` is non-trivial. The
  existing `hit_test_node` recurses depth-first and returns `Option<WidgetId>` on the
  first hit (early exit). Collecting the ancestor path requires a different traversal
  pattern: the recursive call must succeed before we can decide to include the current
  node. One correct approach is to pass a `&mut Vec<WidgetId>` accumulator into the
  recursive call, push to it on successful descent, then reverse the result at the top
  level. Another is to return `Option<Vec<WidgetId>>` from the recursive call and prepend
  the current node's ID before returning. The result must be in root-to-leaf order.
  Write the test first (root → mid → leaf 3-level case) to verify ordering before trusting
  the implementation.
- [x] `set_active(&mut self, widget_id: WidgetId)`:
  Set widget as active. Only one widget can be active at a time. Active widget
  receives all mouse events regardless of pointer position.
  If another widget was active, generate `ActiveChanged { is_active: false }` for
  the previous widget before setting the new one.
- [x] `clear_active(&mut self)`:
  Clear active state. Must be called on mouse-up.
- [x] `request_focus(&mut self, widget_id: WidgetId, focus_manager: &mut FocusManager)`:
  Transfer focus. Calls `focus_manager.set_focus(widget_id)` to keep `FocusManager`
  in sync, then generates `FocusChanged(false)` for old focused widget and
  `FocusChanged(true)` for the new widget. Updates `focus_within` for all ancestors
  using the stored `parent_map`.

  **Important**: `InteractionManager` does NOT store a reference to `FocusManager`. It
  accepts `&mut FocusManager` as a parameter only on the methods that need to sync focus
  state (`request_focus`, `clear_focus`). This avoids storing a mutable reference in the
  struct and keeps the ownership model simple. The caller (app layer / Section 03 event
  pipeline) holds both managers and passes them together.

- [x] **Ancestor tracking for `focus_within`**: The manager needs to know widget
  parent relationships to set `focus_within` on ancestors. Use option (a): accept a
  parent map `HashMap<WidgetId, WidgetId>` built during layout. The layout tree already
  has this information and the map is cheap to build once per layout pass. Do not use
  the Section 02 widget-path result for this purpose — that path is the hit-test ancestor
  chain for the pointer position and is unrelated to the focused widget's ancestry.

- [x] **Parent map extraction**: Implement `fn build_parent_map(root: &LayoutNode) ->
  HashMap<WidgetId, WidgetId>` in a dedicated `oriterm_ui/src/interaction/parent_map.rs`
  sub-module (NOT inside `manager.rs` — keep `manager.rs` under 500 lines). Declare it
  as `mod parent_map;` in `interaction/mod.rs` with `pub(crate) use parent_map::build_parent_map;`.
  This function performs a depth-first traversal of the layout tree, recording
  `child_widget_id → parent_widget_id` for every node pair where both have
  `widget_id: Some(_)`. `LayoutNode` is forward-only (children `Vec`, no parent pointer),
  so a traversal function is the only way to build this map. Called once after each layout
  pass. `InteractionManager` stores the last-built parent map and uses it in
  `request_focus()` to walk ancestors for `focus_within` updates. The
  `set_parent_map(&mut self, map: HashMap<WidgetId, WidgetId>)` method on
  `InteractionManager` stores this map.

- [x] **Tab-key focus cycling bridge**: `FocusManager::focus_next()` / `focus_prev()` are
  called from the app-level keyboard handler when Tab/Shift+Tab is pressed. After
  `FocusManager` advances its internal `focused` field, the app layer must call
  `InteractionManager::request_focus(new_id, focus_manager)` to keep both managers in sync.
  Add a checklist item for this wiring in Section 03 (event pipeline) or Section 08
  (migration). For Section 01, `InteractionManager` must NOT call `FocusManager::focus_next`
  or `focus_prev` — it only calls `set_focus` / `clear_focus` on the `FocusManager`
  parameter.
- [x] `clear_focus(&mut self, focus_manager: &mut FocusManager)`:
  Remove focus from current widget. Calls `focus_manager.clear_focus()` to keep sync.
  Accepts `&mut FocusManager` parameter (same ownership model as `request_focus`).
- [x] `get_state(&self, widget_id: WidgetId) -> &InteractionState`:
  Returns interaction state for querying in paint/layout contexts.
  Returns a static `InteractionState::new()` (all false) for unregistered widgets
  instead of panicking. This handles edge cases during widget tree rebuilds.
- [x] `drain_events(&mut self) -> Vec<LifecycleEvent>`:
  Returns and clears pending lifecycle events for delivery.
- [x] **`FocusManager` co-existence summary**: The existing `FocusManager` in
  `oriterm_ui/src/focus/mod.rs` retains tab-order cycling (`focus_next` / `focus_prev`).
  `InteractionManager` subsumes focus-state tracking (`focused_widget`, `focus_within`).
  The two are kept in sync via parameter passing: `request_focus` and `clear_focus` on
  `InteractionManager` each accept `&mut FocusManager` and call `set_focus` /
  `clear_focus` on it. `InteractionManager` does NOT store a `FocusManager` field.

## 01.3 Lifecycle Events

**File(s):** `oriterm_ui/src/interaction/lifecycle.rs`

Lifecycle events fire automatically when interaction state changes. These replace the
current synthetic `HoverEvent::Enter/Leave`.

- [x] Define `LifecycleEvent` enum:
  ```rust
  pub enum LifecycleEvent {
      /// Hot state changed. Fires BEFORE the triggering MouseMove (Druid pattern).
      HotChanged { widget_id: WidgetId, is_hot: bool },
      /// Active state changed (mouse capture acquired/released).
      ActiveChanged { widget_id: WidgetId, is_active: bool },
      /// Keyboard focus changed.
      FocusChanged { widget_id: WidgetId, is_focused: bool },
      /// Widget added to tree (for initialization).
      WidgetAdded { widget_id: WidgetId },
      /// Widget removed from tree (for cleanup).
      WidgetRemoved { widget_id: WidgetId },
  }
  ```
- [x] Lifecycle events are queued by `InteractionManager` and delivered to widgets
  via a new `Widget::lifecycle()` method (see Section 08).
- [x] `HotChanged` must fire before `MouseMove` is delivered to the target widget.
  This ensures paint logic sees the correct hot state when processing the move event.
- [x] Add `WidgetDisabled { widget_id: WidgetId, disabled: bool }` variant to
  `LifecycleEvent`. This fires when `InteractionManager::set_disabled()` is called so
  widgets can perform cleanup on disable (e.g., clearing `pressed` state, cancelling an
  in-flight drag). Without it, a widget disabled mid-drag will still show the pressed visual
  state on the next repaint. The delivery mechanism (passing the event to `Widget::lifecycle()`)
  is Section 08's responsibility; define the variant here so it is available.

- [x] **`EventCtx::for_child()` replacement**: The current `for_child(&self, child_bounds,

**Exit Criteria:** Unit tests demonstrate that moving a pointer over a nested widget tree
produces correct `HotChanged` events in the right order, with ancestors getting
`hot=true, hot_direct=false` and the deepest widget getting `hot_direct=true`.
Additionally: `deregister_widget` on the currently-hot widget clears hot path and emits
`HotChanged(false)` for all ancestors that were hot; `set_active` on widget B while widget A
is active emits `ActiveChanged(false)` for A then `ActiveChanged(true)` for B; `focus_within`
is set on all ancestors between root and the focused widget after `request_focus()`.
